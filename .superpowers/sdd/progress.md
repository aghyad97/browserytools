- ~~T4 minor: for a table with NO outer border, `box` is the outermost INNER rules~~ RESOLVED by review fix (commit `a206fd0`): `DetectedTable` now carries `segments: Segment[]`, the exact segments consumed. **T5/T6 MUST exclude table text by IDENTITY against that array — never re-derive exclusion from `box`.** `box` is for positioning the block in document order only.
- T4 -> T5/T6 BINDING: `DetectedTable.segments` holds the exact Segment objects consumed. Exclusion MUST be by identity (Set<Segment>), NEVER re-derived from `box` (box can be tighter than the visual table -> data loss + leak). **T5/T6 MUST NOT clone/map Segment objects** — pass references through, or identity exclusion silently fails and table text duplicates into paragraphs. If cloning becomes unavoidable, add a stable `id` to Segment instead.
- T4 -> T12: 0.5x line-height row tolerance and RTL cell/column ordering are pinned only by SYNTHETIC tests — no real fixture. Verify on real documents.
- T4 decision (confirm at T12): 1x1 grids now return null, so a genuine single-cell table is missed. Judged cheaper than yanking bordered callouts/pull-quotes out of paragraph flow.
- T5 -> T6 (API DESIGN, act on this): `classify(regions, tables)` positions each table by finding the region whose segments have the most OVERLAP (by identity, via Set membership) with `table.segments`, then interleaves it among that region's lines by `table.box.y1`. This assumes a table's segments all land in ONE region — true today because `regions` partitions every page segment. If T6 ever calls `orderSegments` on anything less than the full page (e.g. a sub-slice), this majority-vote assignment silently degrades rather than erroring.
- T5 -> T6: font-size statistics (body size + heading-size clusters) are computed from ALL segments across all regions, INCLUDING the ones a table will consume — not from the post-exclusion set. Measured necessity on doc3-tables.pdf: excluding table cells first leaves only the 2 heading-sized lines as input, so the body-size histogram degenerates to a single bucket and neither line clears the 1.12x ratio against itself. Computing stats pre-exclusion (but still emitting text post-exclusion) fixes this and is also the more literal reading of "across all input."
- T5 -> T6: doc1-simple.pdf's list has NO leading marker glyph at all (verified via codePointAt on the raw TextItems) — it's pure 30pt indent relative to the body margin. `classify` therefore treats "a run of >=2 consecutive lines sharing a left indent >=10pt beyond the document's body indent" as a list even with zero marker characters; a marker match alone (viaMarker) is the only way a length-1 run still counts as a list. If a future fixture has genuinely indented body text that ISN'T a list (e.g. a block quote), this heuristic will misfire — no fixture currently exercises that case.
- T5 -> T6: heading merge across region boundaries (constraint 4) only ever looks at STRICTLY ADJACENT items in the flattened, already-globally-ordered sequence (region index primary, y descending secondary) — same clustered font size AND a vertical gap <= 3x the larger line's font size. Two same-size headings separated by any other content (paragraph, list, table, or even one plain line) never merge, by construction, not by the gap check alone.
- T5 -> T12: HEADING_MERGE_GAP_RATIO=1.6 pinned by ONE fixture on the must-merge side (doc2 measured 1.46); MIN_INDENT_ONLY_LIST_RUN=3 matches doc1's only indent-list fixture with ZERO margin below it. Both need real-world verification.
- T5 minor -> T12: font-size stats are computed pre-exclusion over ALL segments incl. table cells (necessary for doc3). On a table-heavy page whose table type differs from body type, the character-weighted body size could skew the 1.12x heading threshold. Check on a real financial-report-style PDF.
- T5 minor: table interleave ORDER within a single region (table vs surrounding prose) is untested — doc3 asserts counts + no leaked text, not relative position. Verify in T12.
- T6 -> T12 (pre-existing from T4, real-world common): `detectRuledTable` builds ONE page-wide grid, so TWO distinct ruled tables on the same page would merge into a single bogus grid (all rules clustered together). Untested — no fixture has two ruled tables on one page. Real documents (reports, invoices, spec sheets) do this often. Verify in T12; if broken, T4's detector needs to segment rules into connected components before building grids.

Running borderless detection on it unfiltered greedily matches the RULED table's grid and never reaches the borderless one. Correct order = detectRuledTable page-level FIRST -> filter its consumed segments by identity -> per-region borderless. dedupeTables is a second independent guard.
